2 相关技术与理论基础

本章对系统开发过程中涉及的核心技术做介绍，以后端语言与框架为支撑、以数据持久化方案为基础、以前端开发平台为入口、以大模型服务为智能核心、以对话推荐理论为设计依据，为后续章节的系统设计与实现提供技术方面的铺垫。

2.1 Python

Python 是一种通用的高级编程语言，由 Guido van Rossum 于 1991 年首次发布，以简洁的语法和丰富的标准库见长，在 Web 开发、数据科学、机器学习以及自动化脚本等方向都有广泛的应用。Python 采用动态类型和自动内存管理机制，开发者可以把注意力放在业务逻辑上，而不用在类型声明和指针操作上多费功夫。

Python 的生态体系比较成熟，第三方包在 PyPI 上的数量已经超过五十万，网络请求、数据库驱动、自然语言处理、图像处理等常见需求都有对应的包可以直接拿来用。对本系统来说，Python 的优势主要体现在几个地方：FastAPI、SQLAlchemy、httpx 等 Web 和数据层库都有稳定的维护和完善的文档，可以缩短开发周期；调用大模型 API、处理 JSON 数据、编写爬虫脚本这类任务在 Python 下做起来比较直接，代码量也相对少；Python 社区在 AI 和推荐系统方向积累了大量实践案例，遇到问题的时候有较多的参考资料可以借鉴。

本系统后端采用 Python 3.11 版本进行开发，以改进的错误提示和性能优化来支撑 AI 问答、推荐计算等计算密集型任务的稳定运行。

2.2 FastAPI

FastAPI 是一个基于 Python 的现代 Web 框架，由 Sebastián Ramírez 于 2018 年底发布，构建在 Starlette 和 Pydantic 之上，兼具异步处理能力和类型驱动的接口定义方式。

FastAPI 基于 ASGI 标准运行，可以在单个进程内处理大量并发请求，适合需要调用外部大模型 API、等待远程响应的场景，与传统的同步框架相比，异步方式可以在等待 I/O 的时间片内处理其他请求，从而提高服务器的吞吐能力。接口定义方面，FastAPI 会根据函数的类型注解自动生成 OpenAPI 规范文档，并提供 Swagger UI 和 ReDoc 两种交互式文档界面，开发者在定义接口的同时就得到了可用的调试页面，省去了手动编写接口文档的功夫，也降低了前后端协作时因文档不一致给沟通带来的麻烦。参数校验方面，FastAPI 利用 Pydantic 模型对请求参数做结构校验，字段缺失、类型不匹配、取值越界等情况会在请求进入业务逻辑之前被拦截并返回明确的错误信息，而依赖注入机制则把数据库会话获取、身份校验等公共逻辑从接口函数中剥离出来，使业务代码更加简洁。FastAPI 的路由定义、请求解析、响应序列化等环节都有简洁的写法，一个包含参数校验的接口往往只需要十几行代码即可完成，降低了同类项目（博物馆导览、传统文化教育、地方文旅等）的开发门槛。

本系统选用 FastAPI 作为后端 Web 框架，以异步处理能力避免单个慢请求阻塞整个服务，以自动文档和参数校验机制方便开发迭代阶段快速定位接口问题，以简洁的路由定义支撑 AI 问答、推荐、活动管理等多类 API 的快速开发。

2.3 SQLAlchemy ORM

SQLAlchemy 是 Python 社区中使用最广泛的 ORM（对象关系映射）框架之一，最早发布于 2006 年。ORM 的核心思想是在关系数据库的表结构和面向对象的类之间建立映射关系，使开发者可以用 Python 类和对象来操作数据库，而不用直接编写 SQL 语句。

SQLAlchemy 提供了两种使用模式：Core 层面向 SQL 表达式，开发者用 Python 表达式构建 SQL 语句，对查询逻辑有更细粒度的控制；ORM 层基于声明式映射（Declarative Mapping），每个数据库表对应一个 Python 模型类，表的列对应类的属性，表之间的关联关系通过 relationship 来描述。ORM 层在业务代码中使用更为普遍，因为它把数据操作封装成了对象方法调用，代码的可读性和可维护性都比较好。SQLAlchemy 的 Session 机制是数据操作的入口，管理着对象的生命周期和事务边界，开发者在 Session 中执行查询、新增、修改操作，最后通过 commit 提交事务，把事务管理从业务代码中抽离出来，避免了手动管理连接和提交的繁琐。

本系统采用 SQLAlchemy 的声明式映射方式来定义数据模型，运行时通过 Session 完成数据操作。选用 SQLAlchemy 的原因在于，非遗平台涉及用户、内容、活动、讨论、推荐日志、AI 问答记录等多种实体，用 ORM 统一管理这些模型的字段定义、索引和关联关系，比手写 SQL 更容易维护，也方便后续根据需求变化调整表结构。

2.4 数据库技术

2.4.1 关系型数据库与 SQLite

关系型数据库以表格形式组织数据，通过 SQL 语言进行查询和操作，是当前应用开发中最主流的数据存储方案。常见的关系型数据库有 MySQL、PostgreSQL 和 SQLite 等几种，它们在功能完整度、并发能力和部署方式上各有侧重。

SQLite 是一种嵌入式关系型数据库引擎，由 D. Richard Hipp 于 2000 年发布。与传统数据库需要独立的服务进程不同，SQLite 把整个数据库存储在一个磁盘文件中，应用程序通过链接库直接读写该文件，不需要安装和配置数据库服务，所以特别适合小型应用、原型开发和单机部署场景。SQLite 支持大部分 SQL 92 标准的语法，包括事务、视图、触发器和子查询等，并发方面采用文件级锁机制，支持多读单写模式，对于中小规模的 Web 应用来说可以满足基本的并发需求，零配置、零维护的特点也降低了部署和运维的复杂度。

2.4.2 数据库选型依据

本系统选用 SQLite 作为数据库，主要是从部署、性能、配合这几个方面来考虑的。部署上，非遗数字传播平台在开发和演示阶段需要频繁重建环境，SQLite 的单文件存储方式不需要额外安装数据库服务，复制数据库文件即可完成数据迁移，降低了环境搭建的时间成本。性能方面，本系统的用户量预期在百人到千人级别，并发写入压力不大，SQLite 的读写性能完全可以覆盖，对于以内容浏览和 AI 对话为主的交互模式，数据库操作以查询为主，SQLite 的查询速度在数据量适中的情况下与客户端-服务器架构的数据库差距不大。配合上，SQLAlchemy 对 SQLite 有完善的支持，连接串只需指定数据库文件路径即可，不需要配置主机、端口和认证信息，简化了开发配置。

当然 SQLite 也有局限性，比如不支持高并发写入、不提供网络访问能力等等。如果后续系统需要部署到多实例或者高并发场景，可以切换到 PostgreSQL 或 MySQL，由于使用了 SQLAlchemy ORM 做数据访问层，切换数据库引擎只需要修改连接串，业务代码无需改动。

2.5 微信小程序

2.5.1 小程序运行机制

微信小程序是腾讯公司于 2017 年推出的一种轻量级应用形态，用户无需下载安装即可在微信内使用。小程序的运行架构采用双线程模型：视图层（WebView）负责页面渲染，逻辑层（JsCore）负责业务逻辑处理，两层之间通过微信客户端的消息通道进行通信，把界面渲染和脚本执行隔离开来，避免了 JavaScript 执行阻塞页面渲染的问题，也提升了运行安全性。

小程序的开发语言由三部分组成：WXML 是标记语言，用于描述页面结构，支持数据绑定、条件渲染和列表渲染等指令；WXSS 是样式语言，在 CSS 基础上增加了尺寸单位 rpx（responsive pixel），可以根据屏幕宽度自适应缩放，解决不同设备上布局不一致的问题；JavaScript 负责页面逻辑，通过微信提供的 API 调用设备能力（如网络请求、本地存储、音频播放等）。

2.5.2 小程序在本系统中的适用性

本系统前端选用微信小程序原生框架进行开发，主要是从低门槛接入、社交传播、开发定制能力这几个角度来考虑的。

低门槛接入方面，微信小程序依托微信生态，用户扫一扫或者搜索即可打开，不需要到应用商店下载安装包，对于非遗文化传播场景来说，降低使用门槛有助于触达更广泛的受众群体，尤其是对手机操作不太熟练的中老年用户。社交传播方面，微信本身具备分享、转发等社交功能，小程序可以自然地融入用户的社交链路，用户看到感兴趣的非遗内容后可以直接转发给朋友或者分享到朋友圈，有利于内容的二次传播。开发效率与定制能力的平衡上，小程序原生框架虽然不如跨平台框架（如 uni-app）那样可以一套代码多端运行，但在页面定制、动画效果和组件交互方面有更大的自由度，本系统的 AI 对话页面涉及数字人表情联动、推荐卡片展开收起、ASK 选项交互等复杂 UI 逻辑，原生开发可以更精细地控制渲染行为和交互细节，也方便后续在真机上调试和演示。

2.6 大语言模型与提示工程

2.6.1 大语言模型基本原理

大语言模型（Large Language Model，LLM）是一类基于 Transformer 架构的深度学习模型，通过在海量文本数据上进行自监督训练来学习语言的统计规律和语义表示。Transformer 的核心组件是自注意力机制（Self-Attention），可以在生成每个词的时候关注输入序列中的所有位置，从而捕捉长距离的依赖关系。

大语言模型的生成过程是自回归的，即每次根据已生成的上文预测下一个词的概率分布，从中采样得到输出词，逐步拼接成完整的文本，使得模型具备了续写、问答、翻译、摘要等多种语言能力。近年来，随着模型参数规模从数十亿增长到数千亿，以及指令微调（Instruction Tuning）和人类反馈强化学习（RLHF）等训练技术的引入，大语言模型在遵循指令、多轮对话和知识推理方面的表现有了显著提升。在应用层面，大语言模型通常以 API 服务的形式对外提供能力，开发者将系统提示词、对话历史和当前问题组装成消息数组发送给模型接口，模型返回生成的文本，不需要在本地部署模型，降低了硬件门槛，也便于利用服务商持续更新的模型版本。

2.6.2 提示工程

提示工程（Prompt Engineering）是指通过设计和优化输入提示词来引导大语言模型产生期望输出的技术手段。与模型参数级别的微调不同，提示工程并没有修改模型权重，而是通过调整输入文本的结构、内容和约束条件来影响模型的行为。常见的提示工程策略有几种：系统提示词（System Prompt）在对话开始的时候设定模型的角色、语气、知识范围和输出约束，相当于给模型划定了一个行为边界，一个设计良好的系统提示词可以让模型在特定领域内给出更专业、更稳定的回答；上下文注入（Context Injection）把外部检索到的知识片段作为参考信息注入到提示词中，使模型在生成回答的时候可以参考这些内容，而不是完全依赖自身的训练数据，可以有效减少模型编造事实的情况；输出约束（Output Constraint）通过在提示词中明确限定回答的长度、格式、语言风格和内容范围，防止模型输出过长、偏离主题或者做出超出能力的承诺，在面向终端用户的产品场景中尤为重要。

提示工程的优势在于不需要额外的训练成本，调整提示词即可快速迭代模型行为，适合资源有限的工程团队，不过对模型行为的控制是间接的，不同模型对同一提示词的响应可能存在差异，需要针对具体模型进行调试和适配。

2.6.3 语音合成技术

语音合成（Text-to-Speech，TTS）是将文本转换为自然语音的技术，现代 TTS 系统通常基于神经网络，将文本先转换为音素序列，再通过声学模型生成频谱特征，最后由声码器合成音频波形，在中文场景下已经可以生成比较自然的语音，语调和节奏接近真人发音。在 Web 应用中，TTS 的常见实现方式是后端将生成的音频文件保存为 MP3 格式，通过静态资源路径返回给前端播放，对前端的要求较低，只需要调用音频播放接口即可，不需要在客户端进行复杂的语音合成计算，为了减少重复生成带来的耗时，通常会对已合成的文本做缓存，相同内容直接返回已有的音频文件。本系统在 AI 数字人模块中引入了 TTS 语音播报能力，使数字人不只是文字问答界面，还具备语音导览功能，提升了用户的沉浸式体验。

2.7 对话推荐系统理论

2.7.1 对话推荐系统的基本思想

对话推荐系统（Conversational Recommender System，CRS）是一类通过多轮自然语言交互来逐步理解用户偏好并给出个性化推荐的系统。与传统的静态推荐系统不同，CRS 认为用户的兴趣表达往往是不完整的，很多偏好是在交互过程中逐渐形成的，系统需要在信息不足的时候主动提问来获取偏好线索，而不能在缺乏依据的情况下直接给出推荐结果。

CRS 的核心流程可以概括为两个交替进行的阶段：偏好获取（Preference Elicitation），即系统通过提出结构化或半结构化的问题来收集用户在类别、地域、场景等维度上的偏好信息；推荐生成（Recommendation Generation），即当系统判断已收集到的偏好信息足够支撑一次有意义的推荐时，便根据用户画像生成推荐结果并呈现给用户。两个阶段之间的切换时机是 CRS 设计中的关键决策点，过早推荐会导致结果不相关，过晚推荐则会让用户失去耐心。对于非遗文化传播场景来说，用户对非遗项目的了解程度差异很大，有的人可能只知道某个类别名称，有的人则能说出具体的传承人和代表作品，静态推荐难以处理这种差异，而 CRS 可以通过对话来评估用户的认知水平，再决定是继续引导还是直接推荐。

2.7.2 置信度评估与模式划分

CRS 在决定何时从提问切换到推荐时，需要一个量化指标来衡量当前用户画像的充分程度，置信度（Confidence）是常用的评估维度，反映了系统对用户偏好判断的确定程度。

置信度的计算通常综合考虑多个信号来源：显式偏好信号来自用户主动设置或回答的信息，可信度最高但获取频率较低；隐式行为信号来自用户的浏览、点击、停留时长等操作行为，数量较多但噪声也较大，需要做归一化和加权处理；对话信号来自 CRS 提问过程中用户给出的回答，介于显式和隐式之间，既有一定的主动性，又是在系统引导下产生的。根据置信度的高低，系统可以将用户状态划分为不同的模式：低置信度对应冷启动模式，系统以提问为主，快速积累偏好信息；中等置信度对应混合模式，系统在推荐的同时继续追问，边推荐边修正画像；高置信度对应精准模式，系统认为已经比较了解用户，可以直接给出个性化推荐。这种划分方式虽然比不上强化学习策略复杂，但是和实际实现一致，更易解释、调试、验证。

2.7.3 混合推荐框架

混合推荐（Hybrid Recommendation）是指将多种推荐策略融合在一起，以弥补单一策略的不足。常见的推荐策略有基于内容的推荐、协同过滤和基于知识的推荐这几种，每种策略都有其适用的场景和局限。基于内容的推荐通过计算候选物品与用户历史偏好之间的特征相似度来排序，可以处理新物品的冷启动问题，但推荐结果容易局限于用户已知的兴趣范围；协同过滤利用用户群体的行为模式来发现潜在的关联，在数据充足时效果较好，但在用户量和交互记录较少的冷启动场景下表现不佳；基于知识的推荐利用领域知识（如知识图谱中的关联关系）来扩展候选集和生成解释，适合领域知识结构化程度较高的场景。

本系统采用混合推荐框架，将多源信号融合后进行加权排序，以非遗平台用户量有限、标注数据稀缺为现实约束，以推荐链路可复现、可解释为论文和答辩场景的需要，以有限计算资源下的稳定运行和快速迭代为工程要求，所以多信号画像驱动的加权融合排序比离线训练为主的复杂模型系统更符合本项目的实际需求。

2.7.4 知识图谱增强推荐

知识图谱是一种以实体和关系为核心的结构化知识表示方式，其中实体代表现实世界中的对象（如非遗项目、传承人、地区等），关系描述实体之间的语义关联（如"属于"、"位于"、"传承"等）。在推荐系统中，知识图谱可以从两个方面增强推荐效果：候选扩充，即当用户画像或行为信号触发了某个实体时，系统在该实体的关联邻域内查找语义相关的对象补充到推荐候选集中，缓解仅依靠行为共现或关键词匹配时候选覆盖不足的问题，在冷启动和兴趣稀疏的情况下提供更稳定的补充信号；解释增强，即把知识图谱中的关系路径转化为自然语言的推荐理由，回答"为什么推荐这个内容"，将解释信息与推荐结果同时呈现给用户，提高推荐的可理解性和说服力，减少用户对推荐结果的疑虑。

在工程实现上，知识图谱增强模块的定位是辅助模块而非独立的图计算子系统，在推荐和问答过程中按需被调用。对于小规模领域知识（如非遗项目的分类体系和地域归属），轻量级的图谱实现即可满足业务需求，不需要引入复杂的图数据库或图神经网络。

2.8 本章小结

本章对本系统所涉及的核心技术做了介绍。后端方面，以 Python 语言配合 FastAPI 框架提供高效的接口开发能力，以 SQLAlchemy ORM 与 SQLite 数据库构成数据持久层，兼顾开发便利性和部署简便性。前端方面，以微信小程序原生框架的低门槛接入和社交传播便利为非遗内容的触达提供平台基础。AI 能力方面，以大语言模型 API 接入提供自然语言理解和生成能力，以提示工程在不修改模型参数的前提下实现对模型行为的约束和引导，以语音合成技术为数字人增加语音导览功能。推荐理论方面，以对话推荐系统的多轮交互逐步获取用户偏好，以置信度评估与模式划分为提问-推荐的切换提供决策依据，以混合推荐框架融合多源信号进行加权排序，以知识图谱从候选扩充和解释增强两个角度提升推荐质量。以上内容为后续章节的系统设计与实现奠定了统一的技术基础。
