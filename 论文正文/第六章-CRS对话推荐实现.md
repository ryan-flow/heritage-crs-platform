# 第六章 CRS对话推荐系统实现

在前文完成系统架构、功能模块与基础测试说明后，本章进一步对 CRS（Conversational Recommendation System，对话推荐系统）的工程实现进行专门说明。该实现并不追求复杂的大模型自主决策，而是在毕业设计的数据规模和部署条件下，构建一个可运行、可解释、可验证的"ASK—REC—反馈—更新"闭环。

## 6.1 CRS总体实现思路

本系统中的 CRS 由三部分共同组成：

1. **对话策略层**：决定当前是继续提问还是直接推荐；
2. **置信度评估层**：量化系统对用户偏好的了解程度；
3. **推荐执行层**：根据当前模式输出内容、活动、话题三类推荐结果及解释字段。

其总体流程可以表示为：

```text
用户提问 / 用户点击ASK选项
        ↓
   CRS策略决策（ASK or REC）
        ↓
  置信度评估（显式 + 行为 + 对话）
        ↓
 推荐执行（内容 / 活动 / 讨论）
        ↓
 推荐理由、解释字段、图谱增强结果
        ↓
 用户点击 / 浏览 / 报名 / 反馈
        ↓
 行为日志回流，更新画像与置信度
```

该结构的优点在于：
- 对话策略与推荐执行解耦，便于调试；
- 推荐结果带解释字段，便于前端展示和答辩说明；
- 行为回流直接进入画像与置信度计算，形成闭环。

## 6.2 CRS接口设计与调用链

当前 CRS 相关接口集中在`backend/app/api/v1/endpoints/ai.py`中，主要包括：

- `/ai/chat`：AI问答主入口，同时返回回答、策略、ASK、推荐卡片和图谱增强结果；
- `/ai/crs/state`：获取当前会话的 CRS 模式、置信度、维度明细和最近 ASK 时间线；
- `/ai/crs/answer`：提交用户对 ASK 的回答，更新偏好并重算置信度；
- `/ai/crs/reset`：重置当前 CRS 会话；
- `/ai/crs/debug`：返回完整决策链路，用于调试与答辩演示；
- `/ai/crs/stats`：返回 CRS 全局统计与行为闭环统计。

因此，本系统的 CRS 不是停留在算法函数层面，而是形成了可直接被前端和答辩链路调用的一组服务接口。

## 6.3 三维置信度计算

### 6.3.1 置信度维度定义

系统采用三维置信度模型衡量"黑塔对用户理解到什么程度"，三个维度如下：

- 显式偏好维度：用户主动设置的兴趣类目、场景、地区，以及 ASK 回答形成的偏好；
- 隐式行为维度：曝光、点击、浏览、报名、互动等行为形成的偏好信号；
- 对话语义维度：用户在 AI 问答中提及的兴趣关键词、实体和场景特征。

### 6.3.2 置信度计算公式

设：
- \(S_e \in [0,100]\)：显式偏好得分；
- \(S_i \in [0,100]\)：隐式行为得分；
- \(S_d \in [0,100]\)：对话语义得分。

则综合置信度定义为：

\[
C = 0.40\cdot S_e + 0.35\cdot S_i + 0.25\cdot S_d
\]

其中，隐式行为采用轻量加权方式区分不同动作强度，例如：

\[
S_i' = 0.35\cdot N_{expose} + 1.2\cdot N_{click} + 1.6\cdot N_{view} + \lambda_1 N_{register} + \lambda_2 N_{engage}
\]

这里：
- `expose`表示被动曝光；
- `click`表示主动点开；
- `view`表示深度浏览；
- `register`表示活动报名；
- `engage`表示社区互动。

该设计体现了"强行为比弱行为更能说明兴趣"的原则。

### 6.3.3 模式阈值划分

系统根据综合置信度将用户划分为三种模式：

- \(C < 40\)：`cold_start`
- \(40 \le C < 70\)：`mixed`
- \(C \ge 70\)：`precision`

此外，还叠加两类特殊策略：
- 用户明确表达"推荐什么、先看什么"等意图时，优先进入`intent_driven_rec`；
- 连续对话轮次较多但仍处于低置信度时，进入`recovery_ask`重新校准。

## 6.4 ASK-REC状态机实现

### 6.4.1 五种策略

`ai_service.py`中的`_crs_decide()`函数实现了五种核心策略：

1. `cold_start_ask`：冷启动阶段先提问，不推推荐卡；
2. `mixed`：推少量卡片并继续追问；
3. `precision`：直接输出最匹配推荐；
4. `intent_driven_rec`：用户主动求推荐时优先推荐；
5. `recovery_ask`：在低置信度但多轮对话后重新追问。

与传统问答系统相比，这种状态机设计的重点不在"回答得更长"，而在"更快知道该问还是该推"。

### 6.4.2 冷启动策略

当系统判定为`cold_start`时，会优先从 A 组 ASK 模板中选择问题，例如：
- A01：你最想了解哪类非物质文化遗产？
- A02：你更关注哪个地区的非遗？
- A03：你更偏好哪种体验方式？

前端首次进入 AI 页时，如果用户处于冷启动状态，还会自动触发第一道 ASK，以缩短冷启动采样时间。

### 6.4.3 混合策略

当用户进入`mixed`状态时，系统不再只问问题，而是"推荐 + 追问"同时进行。此时：
- 后端会返回 1~2 张 AI 推荐卡；
- 同时返回 B 组追问模板；
- 前端展示推荐结果、当前策略说明和 ASK 选项。

这种模式兼顾即时价值与持续学习，避免系统在中间状态下既不够懂用户、又不给用户任何可行动结果。

### 6.4.4 精准推荐策略

当用户进入`precision`状态后，系统停止不必要追问，直接输出更强匹配的推荐结果。此时推荐理由也会更明确，例如优先使用"因为你喜欢……""精准匹配你的……偏好"等表达，并在需要时附带画像来源说明。

### 6.4.5 恢复策略

若系统在多轮对话后仍然低置信度，说明此前的信息采样效果不佳。此时进入`recovery_ask`，通过 R 组模板重新收集更高价值信息，避免持续在错误方向上推荐。

## 6.5 ASK模板与偏好写回

系统中的 ASK 模板按 A、B、R 三组组织，分别对应冷启动采样、混合阶段追问和恢复阶段纠偏。每个模板包含：
- `attribute`：当前问题采集的属性维度；
- `prompt`：提问文本；
- `options`：可选答案；
- `score_delta`：理论上对置信度的提升贡献。

当用户点击选项后，前端调用`/ai/crs/answer`接口，后端在`process_ask_answer()`中执行以下流程：

1. 记录`CrsAskLog`（含`is_skipped`字段，跳过类回答如"跳过""暂时不用"标记为True且不贡献分数）；
2. 根据`OPTION_TO_PREFERENCE`把选项映射到用户偏好字段；
3. 更新会话已提问属性；
4. 重新计算置信度并返回最新模式。

这一机制将"对话中的选择"转化为"可用于推荐的结构化偏好"，同时通过跳过识别避免无效回答虚增对话维度分数。

## 6.6 推荐执行与解释字段

### 6.6.1 三路推荐

当前推荐执行模块统一由`recommendation_service.py`完成，按对象类型输出三类结果：
- 内容推荐；
- 活动推荐；
- 讨论话题推荐。

不同对象的评分公式不同，但都遵循"匹配分 + 新鲜度 - 惩罚项"的统一逻辑。

### 6.6.2 推荐理由生成

系统通过`_generate_reason()`生成推荐理由。该函数按推荐对象类型和 CRS 模式差异化生成文本，例如：
- 冷启动时承认系统尚在探索，先给出热门或高质量方向；
- 混合阶段说明系统已初步理解用户方向；
- 精准阶段明确指出系统是根据用户偏好做出匹配。

同时，在`precision`模式下，系统还会把"偏好画像""浏览记录""AI提问""报名行为""社区互动"等来源压缩为自然语言理由的一部分，增强可解释性。

### 6.6.3 解释字段结构

除了自然语言理由，系统还使用`_build_explain()`返回四层结构化解释字段：

1. **L2 系统依据**：`match_score`、`novelty_score`、`diversity_penalty`、`final_score`等评分明细；
2. **L2 扩展**：`match_detail`子分项（如heritage_match、scene_match等）；
3. **L3 策略上下文**：`crs_mode`和`crs_mode_label`（当前决策模式）、`strategy_context`（策略说明与画像来源标注）；
4. **L4 图谱推理**：`kg_reason`、`kg_score`、`kg_path_text`、`kg_context`。

前端在默认状态下仅展示一句推荐理由；用户点击"展开解释"后，逐层展示评分明细、策略上下文和图谱路径。这样既照顾普通用户阅读体验，也支持老师在答辩中查看算法依据。

## 6.7 KG增强与CRS联动

本系统中的知识图谱不是单独运行的推荐引擎，而是作为 CRS 的增强器参与两件事：

1. **兴趣扩展**：当用户提到"昆曲""苏绣"等实体时，系统可以沿图谱找相似项目与关联方向；
2. **推荐解释**：把图谱路径附加到 AI 推荐卡的解释字段中。

此外，系统还会将识别出的实体映射到用户偏好类别中，自动补充显式偏好并重算置信度。这使知识图谱真正参与了"提问—理解—推荐"的闭环。

## 6.8 行为回流与闭环诊断

### 6.8.1 行为回流

用户在前端进行的曝光、点击、浏览、反馈、报名等操作会写入行为日志。后端使用这些行为：
- 强化用户画像；
- 更新隐式偏好分；
- 参与 CRS 置信度计算；
- 支撑管理端与调试接口统计。

### 6.8.2 调试与统计接口

为验证 CRS 闭环是否真实生效，系统提供两个关键观测接口：

1. `/ai/crs/debug`：返回用户画像快照、置信度明细、会话状态、ASK 历史、策略决策和推荐卡数量，可用于展示完整决策链路；
2. `/ai/crs/stats`：返回模式分布、ASK 回答率、模板使用情况、平均置信度、冷启动收敛情况和行为闭环统计。

其中，行为闭环统计包括：
- 各类行为分布；
- CTR（排除讨论区假click后的真实CTR）；
- 深度阅读率；
- 满意度；
- 活动报名量；
- 有行为用户与无行为用户的平均置信度差值；
- 行为带来的隐式分变化；
- 近 7 天行为时间线。

此外，`/ai/crs/stats`还提供ASK模板级运营统计（`ask_template_stats`），每个模板含展示次数、回答次数、跳过次数、回答率、跳过率、选项分布、模式升级次数和模式升级率。跳过识别通过`CrsAskLog.is_skipped`字段实现，跳过类回答（如"跳过""暂时不用"）不计入有效回答，避免对话维度分数虚高。

这使 CRS 的效果不只是停留在页面展示，而是能被统计、诊断和运营优化。

## 6.9 冷启动流程实例

新用户进入 AI 页后的典型流程如下：

1. 系统识别当前用户置信度较低，处于`cold_start`；
2. 前端自动弹出第一道 ASK，例如"你最想了解哪类非物质文化遗产？"；
3. 用户点击选项后，后端写入 ASK 日志并更新偏好；
4. 系统重算置信度，若进入`mixed`，则开始返回推荐卡片与追问；
5. 用户点击内容、活动或帖子后，行为日志回流；
6. 系统进一步更新隐式偏好和置信度；
7. 当置信度足够高时，切换到`precision`模式，输出更直接的精准推荐。

这一流程体现了系统"先了解，再推荐；边推荐，边修正；有行为，就更新"的闭环思想。

## 6.10 本章小结

本章从真实代码出发，对 CRS 对话推荐系统的实现进行了说明。系统以三维置信度量化用户理解程度，以 ASK-REC 状态机驱动"问与推"的切换，以推荐理由和解释字段增强可解释性，并通过图谱增强、行为回流、调试与统计接口形成完整闭环。与普通问答系统相比，本系统更强调"理解用户并推动下一步行动"；与普通推荐系统相比，本系统更强调"用对话逐步建立偏好并解释推荐依据"。这为后续实验分析与答辩展示提供了直接支撑。
