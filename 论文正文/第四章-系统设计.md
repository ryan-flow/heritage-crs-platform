4 系统设计

4.1 系统整体架构设计

4.1.1 架构设计原则

本系统设计的主要目的就是创建一个可以主动引导、不断理解用户、自动产生推荐的非遗文化传播平台。为达到以上目的，系统设计的原则如下。

以前后端分离为首要原则，前端主要完成交互展示、状态反馈、推荐结果的承载，后端主要完成AI问答、推荐计算、会话状态和日志的处理；以主线趋同为第二原则，内容页、活动页、讨论页、画像页、管理端都服务于AI导览和对话推荐主线，而不是彼此孤立的；以轻量可解释为第三原则，系统优先选择规则可控、方便调试和答辩说明的设计方案，不会使用和当前数据规模不符的复杂训练框架；以闭环可追踪为第四原则，用户的提问、点击、浏览、报名和互动行为都必须被记录下来，才能反过来影响到后续的推荐；以逐渐加强为第五原则，保证AI数字人和推荐系统叠加知识图谱增强、解释字段和统计观测能力，循序渐进、稳扎稳打。

4.1.2 整体架构

本系统使用微信小程序前端和FastAPI后端相结合的方式构建前后端分离架构，可以分成四个部分。

第一层为前端交互层，主要是微信小程序用户端、管理端的页面。用户端有首页、内容页、活动页、讨论页、AI对话页、个人中心页、偏好设置页等；管理端对内容、活动、讨论、用户数据进行运营管理。

第二层为接口服务层，用FastAPI提供RESTful API来接收前端的请求，并把任务交给核心能力层处理。接口有AI问答接口、CRS状态接口、用户画像接口、内容与活动接口、讨论区接口、登录接口等。

第三层为核心能力层，是本系统设计的重点。本层主要是包含AI服务、推荐服务、本地知识检索、知识图谱增强、联网补充、TTS语音播报等模块。其中ai_service.py负责问答组织、会话管理和ASK-REC决策，recommendation_service.py负责画像构建、置信度计算和三路推荐排序。

第四层为数据与外部服务层，主要是SQLite数据库、本地知识数据、推荐日志数据、CRS会话数据、豆包API、联网搜索、微信登录等组成。本层给系统的可信度、行为闭环、AI能力打下基础，缺一不可。

4.2 推荐模块设计

4.2.1 设计定位

本系统推荐引擎的设计目的不是复现复杂的工业级推荐模型，在目前的数据规模下实现稳定的、可解释的、可联动的AI对话推荐能力。系统使用轻量混合推荐的方式，用规则加权的方法把多源信号融合起来，得到适合非遗传播场景的推荐结果，因地制宜、量体裁衣。

4.2.2 推荐输入设计

推荐系统的输入主要来自三类信息。第一类是用户的长期画像，即偏好类别、场景偏好、地区偏好等；第二类是用户的近期行为，即曝光、点击、浏览、活动报名、讨论互动、AI提问等行为；第三类为当前的上下文，即本轮问题文本和当前访问页面的场景，有首页、内容页、活动页、讨论页和AI页。把这三类输入结合在一起，可以使得推荐结果既能体现用户长期喜欢的内容，也能体现用户当前需要的内容，长短兼顾、远近结合。

4.2.3 用户画像设计

系统画像输出不是简单的标签，是由非遗类别权重、场景偏好权重、地区偏好权重、已浏览对象集合组成的三组兴趣权重和一个已看集合。画像构建的时候，系统会从显性偏好、当前问题、浏览记录、AI提问、活动报名和社区互动等各方面开始累积各种信号。

该种设计使用户画像既能够体现长期兴趣，又能够体现短期上下文，是后面三路推荐排序的基础，承上启下、贯穿始终。

4.2.4 三路推荐设计

系统输出三种推荐结果，即内容推荐、活动推荐、讨论推荐。以内容推荐让用户看得懂，以活动推荐让用户愿意参与，以讨论推荐让用户愿意交流。系统并不把推荐局限于一个对象上，而是从了解、参与、交流这三个方面来组织推荐输出，使AI数字人在对话中给出更完整的下一步行动建议。

4.2.5 排序与解释设计

在排序上系统整体采用的是匹配分、新鲜度或者场景增益、已看惩罚的组合。不同的对象类型尽管具体的细节不一样，但是都遵循着相同的逻辑，即优先推荐和用户画像相匹配、和当前页面场景相契合、并且不是重复浏览的对象。系统给出的推荐理由会是用户可以理解的自然语言，如"基于您对苏绣的兴趣""因为您最近浏览了昆曲相关内容"等，让推荐的过程有据可查、有迹可循。

4.2.6 CRS对话推荐机制设计

CRS模块的主要设计目的是判断"现在应该继续提问，还是应该直接回答"。过早推荐会造成推荐精度不高，长时间提问会降低用户的体验。系统要在这二者之间取得平衡，即效率。

为了达到上述目的，本系统把CRS分为三个连续的环节，即置信度评价、ASK-REC决策和推荐执行。

系统通过三维置信度评估当前对用户的理解程度，包括显式偏好维度、隐式行为维度和对话维度。设显式偏好得分为Se，隐式行为得分为Si，对话得分为Sd，则综合置信度可以表示为：

C = 0.40Se + 0.35Si + 0.25Sd

显式偏好是由用户偏好设置、ASK选项回答所引起的；隐式行为是由曝光、点击、浏览、活动报名、讨论互动所导致的；对话得分是AI提问中表现出的非遗关键词、地区特征、场景偏好。采用该模型的原因有两点，一是它和系统真实的实现是一致的，二是它足够清晰，可以清楚地说明系统为什么从询问转为推荐。

根据置信度高低，系统将用户状态划分为三种模式：cold_start表示对用户了解较少，优先采样高价值偏好信息；mixed表示已有部分画像基础，可以边推荐边追问；precision即系统对于用户的兴趣方向已经比较明确，可以减少打扰，给出更加有针对性的推荐。除此之外，系统还设计了两种特殊的策略，一种是用户主动提出"想要推荐"的时候触发的意图驱动推荐，另一种是在多轮对话之后仍然没有有效偏好的时候触发的恢复式提问。

本系统使用ASK-REC状态机来组织多轮对话。冷启动阶段先用ASK，用少量问题快速得到类别、地区或者体验方式等显性偏好；混合阶段采取"边推荐边追问"的方式，在给出初步推荐结果的时候继续补充画像；精准阶段以REC为主，使用户能很快得到更加契合的内容、活动、讨论推荐。用户明确表示"给我推荐""我想看看有什么"的时候，系统就会停止继续追问，直接给出推荐；如果系统连续多轮都没有得到有效的偏好，那么就会触发恢复式ASK，主动帮助用户重新找到兴趣的方向。

为支持状态机运行，系统设计了三类ASK模板。A组模板重视非遗类目的、地域的和体验方式的高价值属性，冷启动的时候优先使用；B组模板用作混合阶段，围绕当前推荐结果提出追问和微调；R组模板在对话出现偏离或者低效的时候，可以起到重新校准方向的作用。每一个ASK模板都有提问文案、可选项、属性和分值增量。用户点击选项之后，系统就会把ASK日志存起来，把选项映射到用户偏好字段上，然后重新计算会话状态。

4.3 AI数字人设计

4.3.1 角色定位设计

本系统中的黑塔并不是一般的聊天机器人，它是专门用来在非遗传播场景中服务用户的AI数字人导览官。其职责有四个方面。以知识传授为首要职责，围绕中国非遗相关问题进行解释说明，承担导览入口的作用；以推荐引导为第二职责，将用户的提问自然地引向内容、活动、讨论推荐，不让对话停留在静态问答的层次上；以采集偏好为第三职责，经过多轮对话和ASK模板的收集，得到用户对于类别、地区、体验方式等方面的兴趣；以闭环触发为第四职责，把用户在对话、推荐过程中产生的行为的反馈输入到画像、CRS状态当中，使系统对用户的理解越来越深入。黑塔在系统中起到的作用更加接近于智能导览入口和推荐协调器，一专多能、统筹全局。

4.3.2 AI回答链路设计

为了达到回答自然度和知识可信度的目的，在设计时使用的是本地知识优先的AI回答链路。完整的流程为：接收用户问题和上下文推荐卡片；检索本地知识库，判断是否存在高相关非遗知识；读取当前CRS会话和置信度状态；生成推荐载荷，准备AI推荐卡片；根据对话策略决定是优先ASK还是优先REC；若需要语言生成，则调用豆包API完成回答生成或知识润色；把最终的回答、推荐卡片、ASK选项、策略说明和会话状态返回给前端，同时记录日志。

该设计的核心并不是要让大模型直接回答一切，而是把大模型放在一个受约束、可追踪的流程里，有的放矢、收放自如。

4.3.3 提示词约束设计

系统用提示词工程和上下文注入对输出加以控制。提示词的设计分为三步。第一层为全局角色提示，确定黑塔身份、语气、服务范围和知识边界，要求回答聚焦非遗场景，不得编造事实，可以自然关联平台已有的内容、活动和讨论。第二层为任务型提示，当本地知识被命中之后，系统就会要求模型对参考信息进行简化的润色，使回答更加口语化，更适合小程序的展示；当本地知识没有被命中的时候，就使用直接回答型提示，限定输出范围并承诺能力。第三层就是CRS感知提示，系统会根据当前用户处于cold_start、mixed还是precision模式，动态调整提示词语气和深度。冷启动阶段回答浅显易懂、激发兴趣为主，精准推荐阶段可以有针对性的结合偏好方向回答。

提示词在本系统中既回答了"回答是否像人"的问题，又参与到AI数字人和CRS推荐的协同设计当中，一举两得、相辅相成。

4.3.4 AI对话页信息层次设计

为了避免AI页面出现信息过多的问题，本系统把AI页面分成三层信息结构。第一层为任务主线层，包含黑塔数字人形象、对话消息区、输入框、当前模式提示、ASK提问卡片，突出用户正在进行的交互任务。第二层是推荐结果层，主要是显示AI推荐卡、推荐理由、下一步建议，让用户在提问之后可以立刻采取浏览、报名或者进入讨论等行动。第三层为解释层，是置信度的详细部分，也是策略的解释和图谱依据的部分，还有时间线部分。本层不允许普通用户查看，但是系统调试、答辩等场合有很重要的作用。

分层设计既可以满足普通用户即时使用的需要，也可以满足论文展示中需要解释系统的需要，各取所需、互不干扰。

4.4 知识图谱设计

4.4.1 设计定位

知识图谱在本系统中所处的地位不是主排序器，而是一个推荐和AI对话的加强器。其主要作用有两点，一是扩大推荐的候选人范围，二是加强推荐的理由解释，锦上添花、画龙点睛。

4.4.2 图谱增强方式

当用户在问题中提到具体的非遗实体，比如昆曲、苏绣、剪纸等，系统就会借助知识图谱里的相近实体、类别关系和路径关系来拓展出更多的推荐候选。同时系统还可以把路径关系转换成自然语言推荐理由，让用户明白不同的项目之间为什么会相关。

尤其适合于非遗场景，很多用户并不熟悉完整的分类体系，知识图谱可以从中以用户已经提到的对象为起点，向更丰富但是不脱离主题的方向延伸出推荐结果，由此及彼、触类旁通。

4.4.3 CRS与图谱联动设计

图谱增强不是推荐结束之后才出现的，可以在CRS对话中起到影响偏好判断的作用。当系统识别出用户提到具体的实体的时候，就可以将其映射到画像特征里，然后给推荐卡加上图谱理由和路径说明。图谱一方面加强了推荐的解释，另一方面也参与了用户的兴趣建模，一石二鸟、相得益彰。

4.5 内容治理体系设计

4.5.1 设计定位

内容治理体系是保证平台内容质量的防线。非遗内容的来源有爬虫采集和人工录入两种，其中爬虫采集的内容质量参差不齐，需要经过质量评分、审核发布、白名单回补三个环节的筛选才能面向用户展示。

4.5.2 质量评分机制

系统使用QualityScorer质量评分器对每条内容计算0到0.99的质量分。评分规则如下表4-2所示。

表4-2 内容质量评分规则

| 评分维度 | 条件 | 加分/减分 |
|---------|------|----------|
| 标题质量 | 标题长度8~40字 | +0.2 |
| 标题质量 | 标题长度>=6字 | +0.1 |
| 正文质量 | 正文长度>=300字 | +0.35 |
| 正文质量 | 正文长度>=800字 | +0.15 |
| 摘要质量 | 摘要长度30~120字 | +0.1 |
| 多媒体 | 有图片 | +0.05 |
| 元数据 | 有发布时间 | +0.05 |
| 元数据 | 有标签 | +0.05 |
| 地域信息 | 有地区信息 | +0.05 |
| 来源权威 | 含"中国非遗""政府""大学""文旅"等 | +0.1 |
| 不良信号 | 含版权声明、ICP、广告、赞助等 | -0.25 |
| 元数据模式 | 标题匹配"目录""摘要""参考文献"等 | -0.2 |
| 乱码 | 标题乱码 | -0.2 |
| 乱码 | 正文乱码 | -0.15 |

质量分低于0.6的内容在入库时即被过滤，不会进入后续的审核流程。

4.5.3 审核发布机制

内容审核状态有三种：pending（待审核）、approved（审核通过）、rejected（审核驳回）。管理员在管理端对内容进行审核操作，审核通过的时候如果内容处于草稿状态则自动发布，审核驳回的时候内容回退至草稿状态。普通用户可见的内容必须同时满足两个条件：审核状态为approved，且质量分不低于0.8。该双门槛机制保证了面向用户展示的内容在知识准确性和内容完整性上都达到了较高的标准。

4.5.4 白名单回补机制

白名单机制是对已入库内容的二次质量把关。系统定期扫描全量内容，筛选出质量分不低于0.8、审核状态为approved、正文字数不少于280字、标题合规、正文不乱码的内容构建白名单。白名单内的内容自动设为published和approved状态，质量分排名前12条自动加精。非白名单内容取消加精标记。白名单回补机制保证了即使管理员未逐条审核，高质量内容也能自动获得曝光机会，优胜劣汰、去芜存菁。

4.6 数据库设计

4.6.1 概念结构设计

系统的E-R图围绕用户、内容、活动、讨论、推荐和知识六个核心实体展开。用户和内容之间有收藏关系，用户和活动之间有报名关系，用户和讨论话题之间有发帖、评论、点赞、收藏关系，用户和推荐之间有行为日志关系，用户和AI之间有问答记录和CRS会话关系。内容实体有审核状态、质量评分、精选标记等治理属性。推荐日志实体记录了行为类型、推荐对象、来源场景和推荐解释。CRS会话实体记录了当前模式、轮次和已问属性列表。

4.6.2 逻辑结构设计

逻辑结构设计是将概念模型转化为具体的数据库表结构。系统数据库共包含16张数据表，按照业务领域可以划分为用户域、内容域、活动域、社区域、推荐域和知识域六个部分。各表的结构已在第三章系统数据字典中详细列出，此处不再重复。

数据库设计以闭环可追踪为标准，除了用户、内容、活动、讨论等基本业务数据之外，还保留了AI问答日志、推荐日志、CRS会话数据、ASK日志等四类数据。这些数据一起组成了系统之后画像更新、统计分析以及实验验证的数据来源，环环相扣、缺一不可。

4.7 接口设计

4.7.1 接口规范

系统所有接口遵循RESTful API设计规范，统一挂载在/api/v1前缀下。接口规范如下：请求和响应均使用JSON格式；管理员接口使用Bearer Token进行身份校验，用户侧接口使用微信openid标识身份；列表接口支持分页查询，默认每页20条；错误响应统一返回detail字段说明原因；AI对话接口支持SSE流式输出，减少用户的等待感。

4.7.3 关键接口详细设计

（1）AI对话接口

AI对话接口是系统的核心主线能力，设计上不是孤立的问答接口，而是将KB检索、CRS决策、推荐生成、回答润色组织在一个请求链路中。请求参数有user_id、question、scene三个字段。响应结果包含answer（回答文本）、recommend_cards（推荐卡片列表）、ask_options（ASK选项）、crs_state（CRS状态快照）、source（回答来源）五个部分。流式接口使用SSE事件流，依次推送crs_state、content、recommend、done四种事件类型，前端可以逐步渲染，减少用户的等待感。

（2）推荐接口

推荐接口集成了CRS置信度状态，返回结果中每条推荐项都包含reason（用户可读理由）和explain（系统依据JSON）两个字段。reason字段是面向用户的自然语言解释，如"基于您对苏绣的兴趣推荐"；explain字段包含match_score、match_detail、crs_mode、strategy_context等子项，供系统调试和论文展示使用。

（3）内容审核接口

内容审核接口是内容治理体系在接口层的体现。管理员调用审核接口时传入review_status字段（approved或rejected），系统自动执行状态联动：审核通过的时候若内容处于草稿状态则自动发布，审核驳回的时候内容回退至草稿状态。同时系统还提供了内容审计报告接口和待审核内容列表接口，方便管理员批量处理。

4.8 本章小结

本章围绕系统实现主线，对系统架构、推荐模块、AI数字人、知识图谱、内容治理体系、数据库和接口组织方式进行了设计说明。以架构设计原则为统领，以推荐模块和AI数字人为双核心，以知识图谱和内容治理为两翼，以数据库和接口为支撑，把对话理解、推荐执行、解释生成、行为回流组织成一个连续的闭环。本系统不是把AI、推荐、内容展示简单地拼接起来，而是在黑塔数字人这个统一的入口下，让各模块各司其职、协同配合，形成了一个有机的整体。
